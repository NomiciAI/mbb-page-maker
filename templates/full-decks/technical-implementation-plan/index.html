Technical implementation archetype
Build answer

The implementation answer should name what to build first and what must be standardized

1Anchor to outcomes

Start from the workflow, decision, or platform outcome the build must enable.

2Define shared layers

Separate reusable architecture from use-case-specific logic.

3Control dependencies

Make system, data, security, and team dependencies explicit.

4Release in gates

Use acceptance criteria and rollback paths before scaling.

5Measure quality

Track performance, adoption, risk, cost, and supportability.

6Close decisions

End with architecture choices, resource asks, and risks to resolve.

Release model

A release model should show gates, evidence, and ownership before scale

1Scope

Lock workflow, users, systems, and success measures.

2Build

Implement core path, integration contracts, and controls.

3Validate

Test quality, risk, performance, security, and adoption signals.

4Release

Launch with observability, support path, and rollback criteria.

5Scale

Promote reusable assets and expand to adjacent workflows.

Roadmap

The roadmap should balance near-term proof with reusable implementation assets

0-6 weeksFoundation

Finalize scope, owners, architecture decisions, and acceptance gates.

6-12 weeksPilot release

Build core workflow and validate integration, data, and risk controls.

3-6 monthsScale wave

Expand users and convert pilot assets into reusable platform patterns.

6+ monthsOperate

Embed runbooks, monitoring, release cadence, and continuous improvement.
