# Technical Implementation Plan

## Summary

Technical implementation full-deck archetype for turning a strategy or architecture decision into buildable releases, integration choices, governance, and risks.

## Visual traits

Blue executive technology style, implementation sequence, architecture stack, dependency/risk register, and roadmap-to-decision close.

## Use when

Use for technical implementation plans, architecture rollout, platform modernization, integration plans, or delivery governance decks.

## Do not use when

Do not use for vendor selection, workshop facilitation, or investment decisions unless the primary audience needs implementation sequencing.

## Storyline pattern

Cover, executive build answer, implementation principles, target architecture, integration/data plan, release model, governance/risk, roadmap, and decisions.

## Layout + component stack

Uses `numbered-list-grid`, `framework-map`, `process-flow`, `comparison-table`, `risk-matrix-register`, `phase-roadmap`, and `decision-log`.

## Theme / assets

Theme: `assets/themes/blue.css`. Optional generated cover asset: `assets/media/covers/strategy-architecture-cover.jpg`. Interior visual support can use `visual-architecture-stack` when the message is about platform layers.

## Source inspiration

Source-neutral implementation planning and architecture rollout patterns. No external deck, brand system, or consulting-firm identity is copied.

## Evidence shape

Best with target-state requirements, system layers, dependencies, release phases, owners, risks, and architecture decisions.

## Path

`templates/full-decks/technical-implementation-plan/index.html`

## Related showcase patterns

`templates/showcase/roadmap-risk-decision-page.html`, `templates/showcase/evidence-table-implication-page.html`, `templates/showcase/steerco-progress-decision-page.html`

## Flexible swaps

Swap architecture stack pages for vendor/platform comparison when choices are unresolved, replace risk matrix with decision-log when blockers are governance-led, and replace roadmap with milestone-track for status decks.

## When to diverge

Diverge when the user needs a technical architecture explanation, vendor choice, or PMO status update instead of an implementation plan.

## Do not copy page order/content

Do not copy the archetype page order, placeholder claims, footer text, or implementation labels. Use it only to understand build-plan pacing and component combinations.
